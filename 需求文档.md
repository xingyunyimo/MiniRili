# MiniRili（迷历）需求与设计文档

## 一、项目概述

### 1.1 产品定位
一款轻量化、离线优先的安卓日历应用，集成万年历、记事提醒、节假日查询等核心功能，不依赖云服务，注重启动速度、数据安全和提醒可靠性。

### 1.2 设计原则
- **离线优先**：核心功能无需联网，节假日/黄历数据打包内置
- **轻量化**：APK 体积小、内存占用低、启动快
- **可靠性**：提醒不漏触达，数据可备份迁移
- **可访问性**：适配大字体、TalkBack、高对比度
- **高效运行**：高效运行，不占用过多存储与内存资源

## EXT-01 天气模块（详细）

### EXT-01.1 背景与目标

在现有日历 APP 上叠加轻量级天气能力，遵循"离线优先、轻量化、零成本起步"原则：
- 数据源：Open-Meteo 免费公开 API（无需 Key，10k 次/天配额）
- 包体积零新增依赖（HttpURLConnection + org.json，Android SDK 自带）
- 隐私：位置数据仅发往 open-meteo，不上传任何其他服务器
- 离线：Room 缓存最近一次天气数据，30 分钟新鲜度；无网时展示缓存并标注"可能过期"

### EXT-01.3 数据源

- **主数据源**：Open-Meteo（https://open-meteo.com）
  - Forecast API：`https://api.open-meteo.com/v1/forecast`
  - Geocoding API：`https://geocoding-api.open-meteo.com/v1/search`
  - Air Quality API：`https://air-quality-api.open-meteo.com/v1/air-quality`
  - Archive API：`https://archive-api.open-meteo.com/v1/archive`
- **免费层限制**：10,000 次/天、5,000 次/小时、600 次/分钟；**仅限非商用**
- **商用路径**：$29/月起订阅，或 Docker 自托管 Open-Meteo 开源版
- **V2 补充**：和风天气（官方预警、分钟级降水，免费 1000 次/天）

### EXT-01.4 架构

```
UI (WeatherCard / WeatherScreen)
    │ observes StateFlow
    ▼
WeatherViewModel (@HiltViewModel)
    │
    ▼
WeatherRepository (@Singleton)
    │
    ├─► WeatherCacheDao (Room) ── 30 分钟缓存
    ├─► CityDao (Room) ── 多城市管理
    └─► WeatherDataSource (接口)
          └─► OpenMeteoApi (HttpURLConnection + org.json)
                    │
                    └─► LocationHelper (FusedLocationProviderClient)
```

设计要点：
- `WeatherDataSource` 接口抽象数据源，MVP 实现 Open-Meteo，V2/V3 可无缝接和风 / 自部署
- 缓存 key = `"{cityId}|{today}"`，命中即直接展示，不阻塞 UI
- 超出预报范围（>15 天）自动回退今日，不报错

### EXT-01.5 权限

| 权限 | 类型 | 用途 |
|------|------|------|
| INTERNET | 静态 | 访问 open-meteo.com |
| ACCESS_NETWORK_STATE | 静态 | 判断网络可用性 |
| ACCESS_FINE_LOCATION | 动态 | 精确定位（仅天气功能使用） |
| ACCESS_COARSE_LOCATION | 动态 | 粗定位（仅天气功能使用） |

隐私合规：位置数据仅发往 open-meteo 获取对应区域天气，不用于其他目的或与第三方分享；城市记录仅本地存储。

### EXT-01.6 缓存与配额管理

- 本地缓存 30 分钟，过期才发请求
- 默认刷新频率 1 小时（日均 ≤ 24 次/城市）
- 添加 5 个城市日均 ≤ 120 次，远低于 10,000 次免费配额
- 配额耗尽降级：切换本地缓存 + 延长刷新间隔

### EXT-01.7 已知限制

- Open-Meteo 免费层仅限非商用，未来商用需订阅或自托管（当前决策：个人使用走公开 API，商用时自部署）
- 无中国官方气象预警（V2 接和风补充）
- 无分钟级降水 API（V2 接和风补充）
- 生活指数需本地规则引擎计算（Open-Meteo 无直出指数）

## 七、性能优化策略

### 7.1 APK 体积优化

| 优化项 | 措施 |
|--------|------|
| 代码压缩 | 启用 `shrinkResources`、`minifyEnabled true`，移除未使用资源 |
| 依赖精简 | 只引入必需库（农历库使用轻量级 MIT 许可证版本） |
| 资源分割 | 节假日数据单独做成资源包，支持按需下载 |
| 资源优化 | 移除多分辨率图片资源，使用 tflite 图片或矢量图标 |
| 多 dex | 限制 `maxSdkVersion` 优化，避免 API 28+ 出现过多 dex 文件 |

### 7.2 内存占用优化

| 优化项 | 措施 |
|--------|------|
| Room 数据库 | 按年月懒加载，历史数据按需查询；使用单例 Database 实例 |
| 图片缓存 | 使用 LruCache 限制缓存大小，及时释放不使用图片 |
| Compose 优化 | 使用 `LazyColumn` 替代 `Column`，避免不必要的重组 |
| 对象池 | 重复使用的对象（如日期格式化器）使用静态对象池 |
| 监控 | 使用 Android Profiler 监控内存泄漏 |

### 7.3 后台运行优化

| 优化项 | 措施 |
|--------|------|
| 提醒调度 | 使用 WorkManager + AlarmManager 混合调度，兼容 Android 10+ |
| 后台服务 | 避免常驻 Service，使用前台服务仅在被通知时启动 |
| 轮询策略 | 放弃定时轮询，改为事件驱动 + 触发式提醒 |
| 通知延迟 | 提醒通知延迟 1-2 分钟发出，减少频繁唤醒 |
| 进程优先级 | 设置 `android:process` 标签，避免主进程被杀；使用 `priority` 优化 |

### 7.4 存储优化

| 优化项 | 措施 |
|--------|------|
| 数据库文件 | 使用 Room，数据库文件自动管理，无需手动优化 |
| 节假日数据 | 放在私有存储，不放在 APK 资源中 |
| 事件附件 | 限制附件大小（单个 ≤ 5MB），过多附件提示压缩/删除 |
| 缓存清理 | 提供手动清理缓存功能，用户可随时清除 |

## 九、风险与合规

### 8.1 侵权风险

| 风险项 | 风险等级 | 规避方案 |
|--------|---------|---------|
| 天气 API（EXT-01） | 🔴 高 | 无免费商用 API，仅做接口封装，由用户自行申请 Key |
| 黄历宜忌文案（CAL-07） | 🟡 中 | 使用开源农历库 + 自写宜忌文案，避免复制商业日历表述 |
| 节假日卡片设计 | 🟡 中 | 自行整理 JSON，不复制其他 APP 视觉设计 |
| 农历算法库选择 | 🟡 中 | 仅使用 MIT / Apache 2.0 许可证，禁用 GPL |
| .ics 格式（DAT-02） | 🟢 低 | RFC 5545 开放标准，无风险 |
| 系统日历互通（DAT-03） | 🟢 低 | 标准 Android API，无风险 |

### 8.2 合规要求

1. **隐私政策（强制）**
   - 首次启动必须弹出隐私政策并获取用户同意
   - 声明：不收集个人信息、数据本地存储、无网络权限（或仅天气功能需要）
   - 上架应用商店（华为/小米/OPPO/vivo/Google Play）的必需项

2. **网络权限声明**
   - 核心功能不申请 INTERNET 权限，增强用户信任
   - 若后续接入天气 API，单独申请并在隐私政策中说明

3. **应用商店审核**
   - 华为/小米/OPPO/vivo：日历类 APP 需通过"应用类型审核"，部分商店要求提供《数据处理协议》
   - Google Play：若使用生物识别（指纹锁），需遵循 BiometricPrompt API 规范

4. **开源许可证合规**
   - 在"关于"页面列出所有依赖库及其许可证
   - 禁用 GPL 类许可证（除非愿意开源整个项目）

### 8.3 技术选型风险提示

- **农历计算**：选择开源库时注意许可证类型（MIT/Apache2 安全，GPL 有传染性）
- **节假日数据**：国务院发布的节假日安排属于政府信息，可自由使用，但数据格式自主设计
- **提醒机制**：AlarmManager 在 Android 10+ 有限制，建议结合 WorkManager 实现跨版本兼容

### 10.13 v1.2 反馈修复批 2 + v1.3 提醒链补强（2026-07）

1. **repeatType 丢失**（P0 bug）：`EventDetailScreen.buildEvent()` 构造 `EventEntity` 时漏传 `repeatType` / `skipDates`，默认值都是 `"none"` / `""`。结果：
   - 创建"每天重复"事件 → 实际写入 none，只预约单次提醒；
   - 编辑已有每天事件 → 保存把 repeatType 洗成 none，UI 显示"不重复"，后续不再触发。
   - 修复：state 变量新增 `skipDates`，`LaunchedEffect` 加载事件时同步，`buildEvent()` 构造时补 `repeatType = repeatType, skipDates = skipDates`。

2. **周期事件不再只预约 6 天**（v1.3 日常"永远每天"需求）：增加 rolling-window 续约机制：
   - `AlarmReceiver.onReceive` 在处理完 eventId > 0 的闹钟后，如果 `event.repeatType != "none"`，调用 `RecurringReminderScheduler.scheduleNextOccurrence(event, currentDateStr)` 取消该 event 未触发的预约并以当前日期为基重新预约下一轮。
   - 这样除非用户主动改 repeatType 或删除事件，闹钟永远按 repeatType 持续触发。
   - `RecurringReminderScheduler.rescheduleAllReminders` 也一并补上一次性提醒分支（之前只处理周期，未处理单次）。

3. **ReminderBootReceiver 空实现**（P1 bug）：`rescheduleAllReminders()` 和 `rescheduleTodayReminders()` 都是空函数体，设备重启 / 时间变更完全不重调度。修复：实现 `runBlocking { recurringScheduler.rescheduleAllReminders() }`，BOOT_COMPLETED 和 TIME_CHANGED 都走完整重调度（包括 DATE_CHANGED）。

4. **锁屏停闹钟失败**（regression）：`AlarmReceiver` 之前没有独立停闹钟路径，只依赖 MainActivity 上注册的 `AlarmScreenOffReceiver`（activity 销毁时注销）。APP 在后台响铃时锁屏 → MainActivity 可能已被回收 → receiver 不存在 → 铃不止。
   - 修复：`AlarmReceiver.playAlarmSound` 期间在 `applicationContext` 上动态注册内部 `ACTION_SCREEN_OFF` 监听，app 进程内 Activity 状态无关；铃声停止或 30s 超时后注销。

5. **stop vs play race**：`playAlarmSound` 在 IO 协程里构造 Ringtone 并写入静态 `activeRingtone`。`stopAlarm()`（Activity.onResume、Notification点击等）可能在写入前执行 → `activeRingtone == null` 关不掉 → 协程写入后放铃 → 声音失控。
   - 修复：加 `@Volatile stopRequested` 标志；`stopAlarm()` 先置标再清引用；`playAlarmSound` 启动前和写入后都检查此标志，若已请求停则不播放并立即清理。

6. **反地理编码双通道 fallback**：Android `Geocoder` 在很多设备（模拟器、国行无 Play Services）上返回空列表，之前上层直接 fallback `"当前位置"`，用户看到的是占位字符串而非真实地名。修复：`LocationHelper.reverseGeocode` 双通道
   - 优先 Android 本地 Geocoder（零外发）；
   - 失败时 Nominatim (OpenStreetMap) 网络反查，解析 `address.state/city/county` 拼成"省 市 区县"；
   - 都不成功 fallback "当前位置"。

### 10.19 闹钟静音修复（2026-07）

> **P0 Bug**：事件设置了"通知栏 + 闹钟"双通道，触发时通知正常弹出但闹钟铃声不响。

- **根本原因**：`AlarmReceiver` 伴生对象 `stopRequested: Boolean` 标志一旦写入 `true` 就**再未重置为 `false`**。
  - `MainActivity.onCreate() / onResume() / onNewIntent()` 每次都调 `AlarmReceiver.stopAlarm()` → 写入 `stopRequested = true`
  - `AlarmReceiver.playAlarmSound()` 入口处检查 `if (stopRequested) return` → 直接返回，不播放铃声
  - 通知走 `NotificationManager`（不依赖此标志），所以用户看到通知但听不到闹钟
  - `stopRequested` 是 `companion object @Volatile` 变量，存活于进程内存，用户打开一次 APP 后所有未来闹钟都被静音
- **修复**：`playAlarmSound()` 入口处先执行 `stopRequested = false`，确保每次新闹钟触发都清除旧状态；`registerScreenOffStopper` 后的二次检查仍保留，防止 reset↔stop 的竞争条件。

### 10.19 桌面小部件刷新重构（2026-07）

> 将原有"事件 5s 循环顺带更新时间"的耦合逻辑拆为三个独立刷新循环，天气预报增加 AlarmManager 后台保活。

1. **时间独立 1 秒刷新**：`Handler.postDelayed` 每 1 秒调用 `Calendar.getInstance()` 读取系统时钟（NTP 同步），`partialUpdateAppWidget` 仅更新 `widget_time` 一个 TextView，不碰天气/事件区域。与事件循环解耦——无论事件数量，时间始终实时走时。生命周期：`onEnabled`/`onUpdate` 启动，`onDisabled` 停止（static Handler 引用防止实例重建后跑飞）。

2. **天气预报 30 分钟双通道刷新**：
   - 主路径：`AlarmManager.setExactAndAllowWhileIdle(RTC_WAKEUP, 30min)` → `ACTION_REFRESH_WEATHER` 广播 → `onReceive` 后台线程全量 `buildDynamicViews` → `updateAppWidget`。进程被杀仍可触发。
   - 兜底：`Handler.postDelayed(runnable, 30min)` 在 APP 存活时提供额外刷新保障，Room 30 分钟缓存防止双通道重复网络请求。
   - 旧 `4×1 WeatherWidgetProvider`（仅依赖系统 `updatePeriodMillis`，刷新不可控）已删除。

3. **事件轮换改为 Handler + partialUpdateAppWidget**：原 `AlarmManager.setExactAndAllowWhileIdle` 改为 `Handler.postDelayed` 5 秒循环，规避 Android 8+ 对短间隔 AlarmManager 的频率限制。`partialUpdateAppWidget` 增量推屏，仅更新事件相关 view（`widget_event_count` / `widget_event_time` / `widget_event_title` / `widget_event_color`），不干扰时间/天气区域。

4. **旧小部件清理**：删除 `CalendarWidgetProvider`（4×4 月历）、`WeatherWidgetProvider`（4×1 天气）及对应的布局文件（`calendar_widget.xml`、`weather_widget.xml`、`widget_calendar_row.xml`）和小部件配置（`calendar_widget_info.xml`、`weather_widget_info.xml`）。Manifest 中的注册条目此前已清理。

| 空气质量 | `air-quality-api.open-meteo.com/v1/air-quality?latitude=&longitude=&current=pm10,pm2_5,ozone,nitrogen_dioxide,sulphur_dioxide,carbon_monoxide&timezone=Asia/Shanghai` | AQI + 6 项污染物 |
| 历史天气 | `archive-api.open-meteo.com/v1/archive?latitude=&longitude=&start_date=&end_date=&daily=temperature_2m_max,temperature_2m_min,precipitation_sum&timezone=Asia/Shanghai` | 回看过去 30 天（WTH-10 暂不做） |

**城市搜索数据源**：本地 `ChineseCityDb`（全国地级市 + 四川区县全覆盖）与 Open-Meteo Geocoding 并发搜索后合并去重，本地数据优先且省份准确。解决了 Open-Meteo 对中国县级行政区覆盖不全、省份标错的问题。

### 12.5 数据架构

```
WeatherRepository (@Singleton)
    │
    ├─► WeatherCacheDao (Room, 30 分钟缓存, key="cityId|today")
    ├─► CityDao (Room, Flow 观察城市变化)
    └─► WeatherDataSource (接口)
          └─► OpenMeteoApi (HttpURLConnection + org.json, 零新增依赖)
                    │
                    └─► LocationHelper (LocationManager)
                            │
                            └─► Geocoder + Nominatim fallback (反地理编码)
```

架构要点：
- `WeatherDataSource` 接口抽象数据源，MVP 实现 Open-Meteo，V2 可接和风天气
- 30 分钟 Room 缓存，过期才发请求；无网时展示缓存并标注"可能过期"
- API 配额耗尽降级：延长刷新间隔 + 本地缓存兜底
- 城市变化全链路自动刷新：`CityDao.observeAllOrdered()` → `WeatherRepository.observeCities()` → `WeatherViewModel.start()` 自动切换首城
